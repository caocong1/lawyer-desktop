## Goal

Add typed frontend API wrappers for all new Tauri commands and update stores with actions that call them. The frontend must be able to load conversations, load messages, delete conversations, restore providers, and switch between conversations.

## must_haves

- All 5 new API wrapper functions in `api.ts` use `invoke()` with correct command names
- `conversation` store gains: `loadConversations`, `loadMessages`, `deleteConversation`, `switchConversation`
- `conversation.store` `createConversation` updated to persist active conversation
- `settings` store gains: `restoreProvider`
- All signals called with `()` — no bare signal references in JSX or logic
- Type imports for `Conversation`, `Message`, `LlmProvider` added where needed
- No circular dependencies between stores

### Task 2: Update conversation store with new actions

<read_first>
src/stores/conversation.ts
</read_first>

<acceptance_criteria>
- `loadConversations` calls `api.getConversations()`, updates conversations signal with result
- `loadMessages(conversationId: string)` calls `api.getMessages(id)`, updates messages signal with result
- `deleteConversation(conversationId: string)` calls `api.deleteConversation(id)`, then: if deleted id matches `activeConversationId()`, set `activeConversationId` to `null` and clear `messages` to `[]`; also remove from conversations list
- `switchConversation(conversationId: string)` sets `activeConversationId`, calls `loadMessages(conversationId)`, calls `api.setActiveConversation(conversationId)`
- `createConversation` updated: after existing creation logic, calls `api.setActiveConversation(newId)`
- All signal writes use setter functions (e.g., `setConversations(...)`)
- All signal reads use call syntax (e.g., `conversations()`, `activeConversationId()`)
</acceptance_criteria>

<action>
Import API functions from `src/services/api.ts` into the conversation store. Add these action functions to the `useConversation()` return object:

1. `loadConversations`: `const result = await api.getConversations(); setConversations(result);`

2. `loadMessages`: `(conversationId: string) => { const result = await api.getMessages(conversationId); setMessages(result); }`

3. `deleteConversation`: `(conversationId: string) => { await api.deleteConversation(conversationId); if (activeConversationId() === conversationId) { setActiveConversationId(null); setMessages([]); } setConversations(prev => prev.filter(c => c.id !== conversationId)); }`

4. `switchConversation`: `(conversationId: string) => { setActiveConversationId(conversationId); await loadMessages(conversationId); await api.setActiveConversation(conversationId); }`

5. In existing `createConversation`, after creating the conversation, add: `await api.setActiveConversation(newConversation.id)`

Ensure all async operations handle errors gracefully (try/catch with console.error).
</action>

### Task 3: Update settings store with restoreProvider

<read_first>
src/stores/settings.ts
src/services/api.ts
</read_first>

<acceptance_criteria>
- `restoreProvider` is an async action in `useSettings()`
- On success with non-null provider: sets active provider fields and `isConfigured` to true
- On success with null: leaves state as-is (not configured)
- On error: logs to console, leaves state unchanged
- Does not trigger UI navigation (that's App.tsx's job)
</acceptance_criteria>

<action>
Import `api.getActiveProvider` into settings store. Add `restoreProvider` async action:

```
const restoreProvider = async () => {
  try {
    const provider = await api.getActiveProvider();
    if (provider) {
      // Set all provider fields from the returned object
      setActiveProvider(provider);  // or set individual fields
      setIsConfigured(true);
    }
  } catch (e) {
    console.error("恢复提供者配置失败:", e);
  }
};
```

Add `restoreProvider` to the return object of `useSettings()`. Adjust the exact setter names to match what exists in the store.
</action>
